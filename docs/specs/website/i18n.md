# Internationalisation (i18n)

> **Version:** 1.0
> **Status:** Stable (M_I18n)
> **Owner:** M_I18n spike (`tasks/archive/0119-spike-i18n.md`)

---

## Overview

The site is published in two locales: **French (canonical)** and
**English (translation)**. The data pipeline remains monolingual
French — analyst and aggregator prompts read FR sources and produce FR
analyses. Translation happens in a single, separate, human-gated
post-processing step on the already-aggregated JSON.

This spec covers:

1. Filename and on-disk convention for translated content.
2. The translatable-fields allowlist and parity validator.
3. The translator prompt and manual ingest workflow.
4. Loader and routing changes in `site/`.
5. Language toggle, fallback policy, and missing-translation behavior.
6. Editorial guarantees and what is explicitly **out of scope**.

---

## 1. Filename convention

| Artifact | FR (canonical) | EN (translation) |
|----------|----------------|------------------|
| Aggregated analysis | `aggregated.json` | `aggregated.en.json` |
| Aggregated draft (pre-review) | `aggregated.draft.json` | `aggregated.en.draft.json` |
| Aggregation notes | `aggregation-notes.md` | *(not translated in v1)* |
| Sources | `sources.md` | *(never translated — see §7)* |
| Raw outputs | `raw-outputs/*.json` | *(never translated — see §7)* |
| Candidate metadata | `metadata.json` | *(never translated — see §7)* |

**Rules:**

- The bare-name file (`aggregated.json`) is **always the FR canonical
  source of truth** — every translation file is a derivative of it.
- Translation files live next to the FR file in the same
  `versions/<date>/` directory.
- The pipeline (`consolidate.ts`, `analyze.ts`, `aggregate.ts`) never
  writes locale-suffixed files. Only `scripts/ingest-translation.ts`
  does.
- Locale codes follow ISO 639-1 lowercase (`en`, `de`, …). v1 ships
  only `en`.

## 2. Translatable fields (allowlist)

Translation **only** changes prose string fields. All structural and
numeric data stays byte-identical. The translation file re-validates
against the **same** `AggregatedOutputSchema` (no new schema_version).

### 2.1 Translatable JSON paths in `aggregated.<lang>.json`

The following JSON paths are the **only** fields whose string values
may differ from the FR source:

| Path | Description |
|------|-------------|
| `summary` | Top-level synthesis (≤2000 chars). |
| `positioning.<axis>.anchor_narrative` | Per-axis (5 axes) interpretive prose. |
| `dimensions.<dim>.headline.text` | Aggregated headline (≤140 chars). |
| `dimensions.<dim>.headline.per_model[*].text` | Per-model raw headline (verbatim from raw outputs — see §2.3). |
| `dimensions.<dim>.summary` | Dimension synthesis. |
| `dimensions.<dim>.problems_addressed[*].summary` | Each problem addressed. |
| `dimensions.<dim>.problems_ignored[*].summary` | Each problem ignored. |
| `dimensions.<dim>.problems_worsened[*].summary` | Each problem worsened. |
| `dimensions.<dim>.execution_risks[*].summary` | Each execution risk. |
| `dimensions.<dim>.key_measures[*].label` | Measure labels. |
| `dimensions.<dim>.risk_profile.<cat>.note` | Aggregated risk note (≤180). |
| `dimensions.<dim>.risk_profile.<cat>.per_model[*].note` | Per-model risk note. |
| `intergenerational.narrative_summary` | Intergen narrative. |
| `intergenerational.horizon_matrix[*].dimension_note` | Per-row note. |
| `intergenerational.horizon_matrix[*].cells.<horizon>.note` | Per-cell note. |
| `intergenerational.horizon_matrix[*].cells.<horizon>.per_model[*].note` | Per-model cell note. |
| `intergenerational.agreement.dissenting_views[*].view` | Dissenting view prose. |
| `counterfactual.narrative_summary` | Counterfactual prose. |
| `unsolved_problems[*].summary` | Each unsolved-problem prose. |
| `downside_scenarios[*].narrative` | Each downside scenario prose. |
| `agreement_map.high_confidence[*].claim_id` | **NOT translated** — claim_id is an identifier. |
| `agreement_map.contested[*].positions[*].position` | Per-model contested-position prose. |
| `agreement_map.notes` *(if present)* | Free-form aggregator notes. |
| `flagged_for_review[*].reason` | Review-flag reason. |
| `aggregation_method.notes` *(if present, free-text)* | Aggregator method notes. |

### 2.2 Non-translatable (must be byte-identical)

- `schema_version`, `candidate_id`, `version_date`
- `source_models[*].provider`, `source_models[*].version`
- `aggregation_method.method`, `aggregation_method.aggregator_model`,
  `aggregation_method.prompt_version`, `aggregation_method.prompt_sha256`,
  `aggregation_method.run_at`
- All numeric fields: `summary_agreement`, `confidence`,
  `positioning.<axis>.modal`, `positioning.<axis>.score_interval`,
  `dimensions.<dim>.grade.consensus`,
  `dimensions.<dim>.grade.dissent.<model>`,
  `risk_profile.<cat>.modal_level`,
  `risk_profile.<cat>.level_interval`,
  `horizon_matrix[*].cells.<horizon>.modal_score`,
  `horizon_matrix[*].cells.<horizon>.score_interval`,
  `intergenerational.agreement.direction_consensus`,
  `intergenerational.agreement.magnitude_consensus`,
  `coverage_warning`
- All identifier arrays: `supported_by`, `dissenters`,
  `agreement_map.high_confidence[*].models`,
  `agreement_map.high_confidence[*].claim_id`
- All `source_refs` arrays (point into FR `sources.md` — never re-numbered)
- All array **lengths** (e.g., `dimensions.<dim>.problems_addressed`
  must have the same number of entries as in the FR file)
- All object **keys** (dimension keys, axis keys, horizon row keys, …)
- Boolean flags: `human_edit`, `coverage_warning`,
  `direction_consensus`

### 2.3 A note on `per_model[*]` text fields

Per-model text fields (headlines, risk notes, horizon-cell notes) are
verbatim copies of what each analyst model wrote. In the FR
canonical file they are FR. In a translation file they are translated
into the target language **with the same fidelity rule as the rest of
the prose** — they are not paraphrased and they are not regenerated
from the raw outputs. They remain attributed to their original
`model` ID. The Transparency drawer always links to the original
`raw-outputs/<model>.json` (FR) so a reader can verify each model's
actual words.

### 2.4 Parity validator

`scripts/validate-translation.ts` (new) takes a path to
`aggregated.<lang>.json` and asserts:

1. The translation parses against `AggregatedOutputSchema` (same
   schema as FR).
2. For every JSON path **not** in the allowlist (§2.1), the value is
   `deepEqual` to the FR file's value at the same path.
3. For every array in the allowlist, the **length** is identical to
   the FR array.
4. `candidate_id`, `version_date`, `schema_version` match exactly.
5. No new keys introduced; no allowlisted keys missing.

Non-zero exit on failure. Used by `ingest-translation.ts` and by the
build-time loader (warning, not failure — see §4).

## 3. Translator prompt + manual workflow

### 3.1 New prompt: `prompts/translate-aggregated.md`

Versioned and SHA256-hashed like every other prompt. It contains:

- The target language and its label (`English`).
- A reproduction of the translatable-paths allowlist (§2.1).
- An explicit ban on:
  - Paraphrase that changes meaning.
  - Adding or removing list items.
  - Translating identifiers, model names, party names, candidate
    names, dimension keys, horizon keys, axis keys.
  - Adding moral/advocacy verbs ("sacrifice", "betray", "save", …).
  - Translating numbers or units (€/year, %, scores).
- A requirement to output a single JSON object that re-validates
  against the same schema, with all non-translatable fields copied
  verbatim from the input.

### 3.2 New scripts

- `scripts/prepare-manual-translation.ts`
  Inputs: `--candidate <id> --version <date> --lang en`. Output: a
  `_translation/` working directory next to `_manual/` with a
  copy-pasteable `translation-bundle.txt` containing the prompt
  followed by the FR `aggregated.json`. Also `.gitignore`'d.

- `scripts/ingest-translation.ts`
  Inputs: `--candidate <id> --version <date> --lang en --input <path>`.
  Behavior:
  1. Parse the input JSON.
  2. Run `validate-translation.ts` parity check.
  3. Write `aggregated.<lang>.draft.json` (never directly the final
     name — preserves the human-review gate).
  4. Append a `translations.<lang>` block to the version's
     `metadata.json` recording: `prompt_sha256`, `prompt_version`,
     `attested_model_version`, `execution_mode`, `ingested_at`,
     `human_review_completed: false`.
  5. Print instructions on how to promote
     `aggregated.<lang>.draft.json` → `aggregated.<lang>.json` after
     human review.

The promote step is a manual `mv`, identical to how `sources.md.draft`
becomes `sources.md`. Same review gate principle.

### 3.3 Metadata schema additive change

`VersionMetadataSchema` gains an optional `translations` field:

```ts
translations: z.record(
  z.string().regex(/^[a-z]{2}$/),  // ISO 639-1 lowercase
  z.object({
    prompt_version: z.string(),
    prompt_sha256: z.string().length(64),
    attested_model_version: z.string(),
    execution_mode: z.enum(["api", "manual_webchat", "copilot_agent"]),
    ingested_at: z.string().datetime(),
    human_review_completed: z.boolean(),
  }).strict()
).optional()
```

Fully additive — existing metadata files keep validating.

## 4. Loader and routing

### 4.1 Loader

`site/lib/candidates.ts::loadCandidate` gains an optional `lang`
parameter:

```ts
loadCandidate(id: string, lang: Lang = "fr"): CandidateBundle
```

Resolution algorithm:

1. If `lang === "fr"`: read `current/aggregated.json` (unchanged).
2. If `lang !== "fr"`:
   a. If `current/aggregated.<lang>.json` exists and parses, return
      it; set `bundle.translation = { lang, status: "available" }`.
   b. Else, fall back to `current/aggregated.json`; set
      `bundle.translation = { lang, status: "missing" }` so the page
      can render a banner.
3. In all cases, run the parity validator against the FR source as a
   build-time **warning** (logged to stderr, does not fail the
   build). Hard enforcement happens in `ingest-translation.ts`.

`bundle.translation.status` values: `"native_fr" | "available" | "missing"`.

`listCandidates()` gains an `availableLocales: Lang[]` field per entry,
derived from the presence of `aggregated.<lang>.json` files. Used by
the comparison page to mark FR-only candidates in the EN locale.

### 4.2 Routing

Three page families gain a `[lang]` route segment. **FR remains
canonical at the bare path; EN lives under `/en/`.**

| Page | FR (canonical) | EN |
|------|----------------|----|
| Landing | `/` | `/en` |
| Candidate | `/candidat/[id]` | `/en/candidat/[id]` |
| Comparison | `/comparer` | `/en/comparer` |

Implementation: a `[lang]` optional catch-all is **not** used (Next.js
static export complications). Instead each page uses an explicit
`[lang]` segment under `/app/[lang]/...` for non-FR locales, and the
FR routes stay where they are. Concretely:

- `site/app/page.tsx` — FR landing (unchanged path).
- `site/app/candidat/[id]/page.tsx` — FR candidate (unchanged path).
- `site/app/comparer/page.tsx` — FR comparison (unchanged path).
- `site/app/[lang]/page.tsx` — EN landing.
- `site/app/[lang]/candidat/[id]/page.tsx` — EN candidate.
- `site/app/[lang]/comparer/page.tsx` — EN comparison.

`generateStaticParams` for `[lang]` returns `[{ lang: "en" }]` in v1
(extensible). The non-FR page modules import the same React
components as the FR modules and pass `lang="en"` down.

Rationale for not introducing `/fr/...`: it would break every existing
URL in the wild. The cost is asymmetry in the route tree, mitigated by
the fact that both paths render the **same components** with a
different `lang` prop.

### 4.3 `lang` propagation in components

- `LangProvider` is removed from `app/layout.tsx`.
- A new `app/[lang]/layout.tsx` wraps the EN tree and sets `<html
  lang="en">` and provides `lang="en"` via `LangProvider`.
- The FR root layout sets `lang="fr"` via `LangProvider`.
- Components keep using `useLang()` as today; only the source of
  `lang` changes (URL-derived, not localStorage).
- `localStorage` persistence is removed; the URL is the source of
  truth.

### 4.4 `LanguageToggle`

Becomes a `Link` (or pair of Links) that swap the URL prefix while
preserving the rest of the path:

- On `/candidat/x` → toggle target is `/en/candidat/x`.
- On `/en/candidat/x` → toggle target is `/candidat/x`.
- On `/comparer?c=a&c=b` → `/en/comparer?c=a&c=b` (query preserved).
- On `/` → `/en`.

If the EN target page would render with `translation.status ===
"missing"`, the toggle still navigates (a banner appears on the
destination page); we do not hide candidates from EN.

## 5. UI strings

`site/lib/i18n.ts` already exists. M_I18n requires:

1. A code audit (grep) for any hardcoded French string in components
   under `site/components/` and `site/app/`. Each is moved into
   `UI_STRINGS` with both FR and EN values.
2. Replacement of every `[EN] …` placeholder in `UI_STRINGS` with a
   real translation.
3. A test (`i18n.test.ts` extension) asserting that every key in
   `UI_STRINGS` has a non-placeholder EN value (`!startsWith("[EN]")`).

Translation guidelines for UI strings:

- Keep idiomatic, neutral English. Avoid Americanisms vs Briticisms
  (use spellings consistent with EU institutional English where
  ambiguous).
- Spectrum labels in `UI_STRINGS.SPECTRUM_*` translate the
  *political vocabulary*, not the party name (already done).
- Section names ("Domaines", "Risques", "Synthèse") translate to
  "Domains", "Risks", "Summary".

## 6. Missing-translation banner

When `bundle.translation.status === "missing"` on a non-FR locale,
the candidate page renders a dismissible banner just under the Hero:

> **English translation not yet available.** This page shows the
> original French analysis. Sources, prompts and raw outputs are
> always French. ([Why?](/en/methode#translation))

Component: `site/components/chrome/TranslationFallbackBanner.tsx`
(new). String lives in `UI_STRINGS.TRANSLATION_FALLBACK_*`.

Comparison page in EN: candidates with missing translations render
their data from FR (no banner per-candidate), and the picker shows a
small "FR" tag next to their name. Landing page in EN: same — FR
fallback for the candidate cards' prose, party/display name unchanged.

## 7. Out of scope (v1)

- Translating `sources.md` — primary-source fidelity is the
  transparency contract.
- Translating `raw-outputs/*.json` — they are the per-model artifact;
  altering them violates editorial principle 5.
- Translating `metadata.json` — party names and candidate display
  names stay verbatim. Spectrum labels are translated via
  `UI_STRINGS`, not via metadata.
- Translating `aggregation-notes.md` — deferred; Transparency drawer
  surfaces a small "available in French only" tag in the EN locale.
- Automated machine translation in CI — every translation is human-
  reviewed before promotion to `aggregated.<lang>.json`.
- More than two locales — schema and routing are extensible, but v1
  ships only `en`.
- RTL languages.
- Locale-aware number formatting on numeric tooltips (use a
  follow-up task once the first non-English non-French locale is
  considered).
- A separate EN `methodology` page beyond the auto-translated UI
  chrome — methodology copy lives in M_Methodology.

## 8. Test strategy

| Layer | Test |
|-------|------|
| Schema parity | `validate-translation.test.ts` — happy path, modified non-allowlisted field rejected, modified array length rejected, modified numeric value rejected, missing key rejected, extra key rejected. |
| Loader | `candidates.test.ts` extension — `loadCandidate(id, "en")` returns translation when present; falls back to FR with `status: "missing"` when absent; FR-only call unchanged. |
| Routing | Build smoke (`npm run build` in `site/`) — both FR and EN trees emit static HTML for landing, candidate, comparison; missing-translation candidate emits an EN page with the fallback banner. |
| UI strings | `i18n.test.ts` — no `[EN] …` placeholders remain; every `UI_STRINGS` key has non-empty FR and EN. |
| Editorial | Snapshot check: an EN candidate page contains no "[FR-only]" residue text outside the banner; party names appear unchanged; numbers appear unchanged. |
